# Skill: TXT 笔记转 Markdown 并自动归档

## 目标

把我输入、记录的一些文本内容转为markdown格式的笔记，并保存到我的笔记目录下并更新每个层级的index文件索引。

- 自动整理为结构清晰、层级分明的 Markdown 格式文件
- 自动判断分类目录
- 自动创建或选择分类文件夹
- 必要时自动生成 index.md 索引 并更新 index.md 索引
- 必要时生成 Mermaid 图
- 使用增强 Markdown 语法（高亮、表格、引用、提示块等）

最终存放路径： `/Users/wms/Documents/mywiki`


---

# 一、分类规则

## 1. 分类来源规则

- 默认读取 txt 第一行作为分类信息. 第一行明确指定分类，例如：

```

docker > docker原理 

```

或

```

docker / dokcer原理

```

则归类到对应目录,不区分大小写。例如我写的是docker,查找匹配的目录不区分大小写，如果目录下有Docker目录，则存入Docker目录下，如果有DOCKER目录，则存入DOCKER目录下，如果没有，则创建docker目录并存入其中。

- 如果第一行没有明确的分类信息，则停止继续执行，并提示用户补充分类信息

- 支持多层目录结构 例如: `Spring > 事物 > 事物的特性` 

那么就需要在【Spring】目录下嵌套[事物]目录，在【事物】目录下创建`事物的特性.md`文件



---

## 2. 分类存在判断

- 若目录存在：
  - 直接存入该目录
  - 更新该目录下的 `index.md` 索引

- 若目录不存在：
  - 创建新目录并将.md文件存入其中
  - 创建 `index.md`
  - 写入索引记录

---

## 3. 目录示例结构

```

mywiki/
├── Spring
│ ├── 调优
| |    |── index.md
| |    |── 调优.md
| |── index.md
|── Docker
| |── docker原理
| |    |── index.md
| |    |── docker原理（1）.md
| |    |── docker原理（2）.md
| |── index.md
```

## 4. 索引文件示例

第一层目录，例如Docker目录下的index.md内容如下：
```markdown
[目录]

1. [Docker原理](docker原理/index.md)
```

第二层目录，例如Docker> docker原理目录下的index.md内容如下：

```markdown
[目录]

1. [Docker原理（1）](docker原理（1）.md)
2. [Docker原理（2）](docker原理（2）.md)
```

索引文件的开头，都以`[目录]`开头，后续是该目录下的所有.md文件的索引链接，按照生成时间顺序递增编号。

# 二、Markdown 生成规则

## 整理要求

txt 内容通常：

- 没有段落
- 没有标题
- 结构混乱
- 无分层

必须整理成：

- 明确的 H1 / H2 / H3 标题
- 清晰的逻辑结构
- 必要时使用列表分层
- 必要时转为表格
- 命令转为代码块或者行内代码`code`
- 关键点加粗`**加粗内容**`
- 重要内容高亮 `==内容==`

---

## 支持使用的 Markdown 语法


支持所有的markdown语法，特别是以下增强语法：

### 1. 基础语法

- 标题 `#`
- 列表 `-`
- 表格
- 行内代码 `` `code` ``
- 块代码
- 引用
- 加粗
- 高亮 `==重点内容==`

---

### 2. 提示块语法（必须灵活使用）

```

::: info
说明信息
:::

::: tip
技巧建议
:::

::: success
成功经验
:::

::: warning
警告提示
:::

::: danger
严重风险
:::

```

---

### 3. GitHub 风格提示块

```

> [!NOTE]
> 内容

> [!TIP]
> 内容

> [!IMPORTANT]
> 内容

> [!WARNING]
> 内容

> [!CAUTION]
> 内容
```

# 三、Mermaid 自动生成规则

如果 txt 内容涉及：

- 系统结构
- 调用流程
- 时序过程
- 架构说明
- 数据流

自动转为 Mermaid 图，例如：

## 结构图

```mermaid
graph TD
A[客户端] --> B[服务端]
B --> C[数据库]
````

## 时序图

```mermaid
sequenceDiagram
用户->>服务: 请求
服务->>数据库: 查询
数据库-->>服务: 返回数据
服务-->>用户: 响应
```

---

# 四、文件命名规则

文件名必须：

* 去除特殊符号
* 使用下划线替换空格
* 中文允许
* 简洁但表达核心主题

示例：

```
Docker_镜像构建流程详解.md
Redis_持久化机制_RDB_AOF.md
```

---

# 五、index.md 自动维护规则

每个分类目录必须有：

```
index.md
```

格式示例：

```markdown
# Docker 笔记目录

1. [Docker镜像构建流程详解](Docker_镜像构建流程详解.md)
2. [Docker网络模式详解](Docker_网络模式详解.md)
```


如果是多层目录，例如`Spring > 事物 > 事物的特性.md` 格式示例:

```
Spring
  |
  +-- index.md
  |
  +-- 事物
  |     |
  |     +-- index.md
  |     |
  |     +-- 事物的特性.md
```

类似wiki通过连接嵌套进入

---

## 若新增笔记：

* 自动追加编号
* 保持递增
* 不重复
* 链接路径正确(可以使用相对路径)

---

# 六、生成内容结构模板

生成的 Markdown 应遵循：

```markdown
# 标题

[生成时间]

> 简要概述

## 目录

内容。。。
```

---

# 七、执行流程

当用户提供 txt 内容时，必须执行：

1. 解析分类
2. 判断目录是否存在, 若不存在 → 创建目录
3. 转成结构化 Markdown 文件，保存在对应目录下
4. 更新每个目录下对应 `index.md`
5. 返回：

* 生成路径
* 文件名
* 是否创建新目录

---

完成后必须告知：

```
已生成 Markdown 文件：

* 分类目录: docker > docker原理
* 新建目录: 否

```

# 九、设计原则

* 清晰优先
* 可读性优先
* 可查阅优先
* 可复用优先
* 结构化优先

简单、清晰、明了

# 补充

1. 每一层文件夹都必须有一个index.md索引文件，记录该层级下的所有.md文件的索引链接，按照生成时间顺序递增编号。
2. 生成的Markdown文件必须包含生成时间戳，方便后续管理和查阅。
3. 生成的Markdown文件必须包含一个简要概述，概括该笔记的核心内容，方便快速浏览和理解。
4. 嵌套层级的index.md索引文件必须正确链接到下一级目录的index.md文件，形成完整的目录结构，方便用户导航和查阅。e.g. `[skills](skills/index.md)`
